## Why

当前的增量更新逻辑存在多个问题：

1. **日期范围设置逻辑错误**：
   - 系统设置了不必要的默认30天、365天等固定范围
   - 没有智能的日期范围计算机制
   - 存在"增量/全量"模式的混淆

2. **盘中数据覆盖问题**：
   - 数据库中最后一条记录可能是盘中请求的未完成K线数据，并非该时间段的真实收盘价
   - 需要智能判断是否应该覆盖，避免覆盖收盘后的真实数据

3. **全资产更新效率低**：
   - 未充分利用Tushare的trade_date机制进行每日批量更新
   - 缺乏针对不同更新频率（日/月/季）的优化策略

4. **更新模式设计不合理**：
   - 当前设计了复杂的"增量/全量/智能增量"模式
   - 实际应该简化为"智能下载/强制覆盖"两种模式

## What Changes

### 核心改进

1. **重构更新模式设计**
   - 移除错误的"增量/全量/智能增量"模式概念
   - 设计两种清晰的模式：
     - **智能下载模式**（默认）：自动检测数据库状态，智能选择全量或增量下载
     - **强制覆盖模式**：忽略数据库状态，根据用户指定范围更新数据
   - 添加 `--force` 参数启用强制覆盖模式

2. **智能下载算法**
   - 对每个symbol查询数据库最新记录时间
   - 如果无记录 → API不传日期参数（全量下载）
   - 如果有记录：
     - 计算从最后记录后一天开始的增量范围
     - 智能判断是否需要覆盖盘中数据（交易时间内覆盖，收盘后不覆盖）
   - 移除所有默认30天、365天等固定范围逻辑

3. **CLI参数增强**
   - 保留并优化 `--dataset` 参数（替代 `--frequency`）
   - 添加 `--force` 参数启用强制覆盖模式
   - 添加 `--trade-date` 参数用于Tushare的每日批量更新
   - 保持 `--frequency` 向后兼容

4. **全资产更新优化**
   - 未指定symbol时，查询数据库所有symbol，执行智能下载
   - 使用Tushare的 `trade_date` 参数进行每日批量更新
   - 支持按日期范围进行定期批量更新（月/季）
   - 不同更新频率采用不同优化策略

5. **盘中数据覆盖判断**
   - 新增 `should_overwrite_latest_record()` 方法
   - 判断逻辑：
     - 如果最后记录不是今天 → 不覆盖
     - 如果最后记录是今天：
       - 交易时间内 → 覆盖（盘中数据会更新）
       - 交易时间外 → 不覆盖（数据已收盘）

6. **更新策略矩阵**
   根据参数组合自动选择最优策略：
   - `symbols` + 无日期 → 智能下载（单symbol）
   - 无 `symbols` + 无日期 → 智能下载（全资产）
   - `symbols` + 日期范围 → 强制覆盖
   - `--force` → 强制全量覆盖
   - `--trade-date` → 批量按日更新（Tushare）

7. **错误处理和容错**
   - 智能模式下的回退机制
   - 数据验证确保完整性

### 涉及的规格

- **cli-update**: 修改现有增量更新场景，增强覆盖判断逻辑
- **data-providers**: 新增增量数据获取抽象方法，实现智能日期范围确定

## Impact

### 受影响的规格
- `specs/cli-update/spec.md` - 修改现有增量更新逻辑
- `specs/data-providers/spec.md` - 新增增量数据获取能力

### 受影响的代码
- `finance_data_hub/providers/base.py` - 新增增量更新抽象方法
- `finance_data_hub/providers/tushare.py` - 实现增量更新逻辑
- `finance_data_hub/providers/xtquant.py` - 实现增量更新逻辑
- CLI update命令实现 - 集成新的增量更新策略

### 预期收益
- **数据质量提升**：避免盘中未完成数据的覆盖问题
- **更新效率提高**：智能的增量更新减少不必要的全量请求
- **开发者体验改善**：统一的增量更新API，降低使用复杂度
- **数据一致性**：所有时间序列数据采用一致的更新策略

### 风险
- 新逻辑引入的复杂度增加，需要充分测试
- 对现有代码的修改可能影响已有功能

## Breaking Changes
- 无破坏性API变更
- 现有CLI命令行为保持兼容，新增智能判断逻辑
